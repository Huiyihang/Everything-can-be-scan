# 万物皆可扫｜优化过程记录

这份文档记录项目从一个初始 PoC 想法逐步变成可手机演示 H5 应用的过程，包括技术路线、功能演进、踩坑点和对应解决办法。它更偏开发复盘，README 更偏运行和交接说明。

## 1. 初始目标

项目主题是“万物皆可扫”：把现实物品变成可探索的趣味入口。

初始 PoC 要求：

- 用户上传一张日常物品照片
- AI 识别物品
- 用该物品第一人称写一篇 100 字以内趣味日记
- 自然植入一个相关平台服务
- 移动端 H5 展示

因此项目重点不是只做“识别准确”，而是同时兼顾：

- 手机端拍照体验
- AI 生成内容的趣味性
- 结果页的产品完成度
- 推荐服务的落地链路
- 本地演示稳定性

## 2. 技术路线选择

### 2.1 前端框架

选择 Next.js App Router + React + TypeScript。

原因：

- 可以同时写前端页面和后端 API 路由
- PoC 阶段不需要额外搭建独立后端
- TypeScript 方便约束 AI 返回结构
- App Router 适合拆分首页、结果页、历史页、服务页

### 2.2 样式方案

选择 Tailwind CSS。

原因：

- 快速搭建移动端界面
- 适合小型 H5 产品快速迭代
- 样式可以跟组件放在一起调整

### 2.3 AI 模型接入

选择阿里云百炼 DashScope OpenAI 兼容接口：

```txt
Base URL: https://dashscope.aliyuncs.com/compatible-mode/v1
Model: qwen3-vl-32b-thinking
```

通过后端 API `/api/scan-object` 调用模型，前端不直接暴露 API Key。

### 2.4 存储方式

PoC 阶段没有引入数据库，选择浏览器本地存储：

- `localStorage`：保存历史记录
- `sessionStorage`：保存当前标签页最近一次扫描结果

这样可以避免早期引入登录、数据库、云存储带来的复杂度。

## 3. 功能演进

### 3.1 项目骨架

先初始化 Next.js 项目，并建立基础目录：

```txt
app/
components/
lib/
```

早期目标是先跑起来，而不是一次性把 UI 和 AI 都接完。

### 3.2 核心数据结构

抽象出 `ScanResult`、`HistoryRecord`、`ServiceRecommendation` 等类型。

这个步骤很关键，因为视觉模型返回的内容天然不稳定，前端展示又需要稳定结构。先定义结构，再让模型输出贴合结构，可以减少后续返工。

### 3.3 基础移动端 UI

实现首页上传、拍照、示例体验入口。

一开始页面偏 Demo，只有按钮和基础布局。后续在移动端体验打磨阶段补充了：

- 更明确的视觉层级
- 玻璃面板
- 颜色系统
- 扫描区域
- 图标按钮
- 移动端按钮尺寸
- 空状态和加载状态

### 3.4 扫描中状态

增加扫描线动效和图片预览。

这样即使模型调用需要几秒，用户也知道系统正在工作，而不是误以为页面卡住。

### 3.5 Mock 扫描流程

在真实 AI 接入之前，用 `mockData.ts` 跑通：

```txt
上传图片 -> 扫描中 -> 结果页 -> 再扫一个
```

这个阶段解决了最核心的交互闭环，后续接模型只是替换数据来源。

### 3.6 结果展示和历史记录

结果页展示：

- 物品名称
- 状态描述
- 第一人称日记
- 情绪标签
- 推荐服务卡片

历史记录保存到 `localStorage`，支持查看最近扫描记录和清空。

### 3.7 后端 API 路由

新增 `/api/scan-object`，把前端和模型调用隔离开：

- 前端只传图片和来源
- API 路由负责校验、调用模型、解析结果、映射服务
- API Key 只在服务端读取

### 3.8 视觉大模型接入

通过 `.env.local` 配置百炼参数。

模型返回后，服务端做结构化解析，并把 AI 输出映射为统一 `ScanResult`。

### 3.9 错误处理和降级体验

加入对以下情况的处理：

- 非图片文件
- 图片过大
- 图片压缩失败
- 网络失败
- 模型调用失败
- 模型返回 JSON 格式异常
- localStorage 写入失败

失败时前端提供：

- 轻量提示
- 重新扫描
- 换张照片
- 查看示例结果

### 3.10 移动端体验打磨

重点优化手机上“像完整产品”的感觉：

- 页面高度适配 `100dvh`
- 按钮触控面积加大
- 图片裁切比例稳定
- 历史空状态更明确
- 扫描区域更有视觉记忆点
- 结果页增加服务推荐和分享入口

### 3.11 推荐服务落地页

最初推荐服务卡只是展示信息，点击不能进入新的页面。

后来新增：

```txt
/service/[serviceId]
```

每个服务页展示：

- 推荐理由
- 3 个模拟商品/服务卡
- 淘宝移动端搜索入口

### 3.12 淘宝搜索跳转

当前采用轻量方案：

```txt
https://main.m.taobao.com/search/index.html?q=关键词
```

这样不需要申请淘宝开放平台，也能演示“从 AI 识物到平台服务”的链路。

### 3.13 历史详情页

历史列表只能看到摘要，后续新增历史详情页：

```txt
/history/[recordId]
```

用户可以回看完整日记、图片、服务推荐和分享图。

### 3.14 图片压缩

手机拍照图片可能很大，直接转 base64 发给模型容易失败。

因此新增前端压缩：

- 控制最长边
- 输出 JPEG
- 降低 base64 体积
- 保留足够识别清晰度

### 3.15 分享结果图

新增 Canvas 分享图能力：

- 结果页可以生成 PNG
- 历史详情也可以生成 PNG
- 支持 Web Share 时调用系统分享
- 不支持时自动下载

### 3.16 独立结果页和刷新恢复

早期结果页存在 `UploadCard` 组件状态里，问题是：

- 刷新页面会丢当前结果
- 手机浏览器返回体验不稳定
- 首页组件承担太多职责

最终改为：

- 扫描成功后保存当前结果到 `sessionStorage`
- 跳转到独立路由 `/result`
- `/result` 页面从 `sessionStorage` 恢复最近结果
- 完整历史仍写入 `localStorage`

这样结果页刷新不丢，首页也更专注于扫描入口。

### 3.17 PWA 基础支持

为了让手机端更接近 App 体验，新增 PWA 基础能力：

- Web App Manifest
- 应用图标
- maskable 图标
- 主题色
- service worker
- 客户端注册逻辑

当前重点是支持“添加到主屏幕”和基础静态资源缓存。完整离线页、安装引导和图标尺寸矩阵可以后续继续完善。

### 3.18 模型请求超时

视觉模型调用可能受网络、图片大小、模型排队等因素影响，长时间不返回会让用户一直停在扫描中。

因此新增两层超时：

- 服务端 DashScope 请求默认 30 秒超时，可通过 `DASHSCOPE_TIMEOUT_MS` 配置
- 前端扫描接口请求默认 45 秒兜底超时

超时后会返回明确提示，用户可以重试或改用示例体验。

### 3.19 按状态优化推荐策略

早期推荐只根据物品类别决定，例如杯子推荐买新杯子、鞋子推荐鞋履护理。这样能跑通链路，但不够聪明：一个“脏杯子”和一个“破杯子”显然不该得到完全一样的推荐。

后续把服务映射升级为“状态优先、类别兜底”：

- `dirty`：优先推荐清洁护理
- `damaged`：优先推荐维修、替换件或安全换新
- `worn`：优先推荐养护、修补和易耗件替换
- `idle`：优先推荐回收、转卖、收纳或旧物改造
- `new` / `used` / `unknown`：继续按物品类别推荐

这样仍然不让 AI 直接生成外链，推荐策略保持可控，同时结果更贴近物品真实状态。

## 4. 踩坑与解决办法

### 4.1 Hydration Error

问题表现：

手机端出现 React hydration mismatch 提示。

原因分析：

SSR 页面和客户端首次渲染内容不一致，常见诱因包括：

- 首屏直接读取 `window`
- 首屏读取 `localStorage`
- 使用 `Date.now()` 或随机值影响首屏结构
- 服务端和客户端渲染分支不一致

解决办法：

- 客户端存储读取统一放到 `useEffect`
- 加入 `hasMounted` / `hasLoaded` 状态
- 服务端首屏先渲染稳定的加载态
- 历史记录、当前结果都在客户端挂载后再读取

### 4.2 结果页刷新丢失

问题表现：

扫描成功后结果存在组件 state 中，刷新页面 state 清空，但历史记录还在。

解决办法：

- 新增独立结果页 `/result`
- 扫描成功后写入 `sessionStorage`
- `/result` 挂载后读取最近结果
- “再扫一个”时清理当前结果并返回首页

### 4.3 手机拍照图片太大

问题表现：

手机原图可能超过十几 MB，转成 base64 后请求体更大，容易触发模型或服务端限制。

解决办法：

- 前端先判断原图大小
- 使用 Canvas 压缩图片
- 限制压缩后 data URL 长度
- 压缩失败或仍过大时提示用户换图

### 4.4 模型返回格式不稳定

问题表现：

视觉模型可能返回解释性文字、Markdown 包裹 JSON，或者字段缺失。

解决办法：

- 提示词中明确要求 JSON
- 服务端解析时提取 JSON 内容
- 对字段做类型校验和兜底
- 格式异常时返回可读错误，不让前端崩溃

### 4.5 AI 推荐链接不可控

问题表现：

如果让模型直接生成服务链接，可能出现不可控 URL、虚假平台或不稳定推荐。

解决办法：

- AI 只负责识别物品和生成日记
- 服务推荐由 `serviceMap.ts` 根据类别映射
- 淘宝关键词也由本地配置生成
- 这样更适合 PoC 演示和后续商业化扩展

### 4.6 localStorage 容量限制

问题表现：

历史记录里包含 base64 图片，保存太多可能超过浏览器存储限制。

解决办法：

- 限制历史记录数量
- 写入失败时保留更少记录重试
- 压缩图片后再保存
- 后续商业化版本应改为云端对象存储

### 4.7 手机局域网访问

问题表现：

电脑能打开 `localhost:3000`，手机打不开。

原因：

`localhost` 指向手机自己，不是电脑；Next.js 默认也可能只监听本机地址。

解决办法：

使用：

```bash
npm run dev -- --hostname 0.0.0.0 --port 3000
```

然后手机访问电脑局域网 IP：

```txt
http://电脑IP:3000
```

必要时允许 Windows 防火墙中的 Node.js 网络访问。

### 4.8 淘宝真实接入复杂度

问题表现：

如果要接真实淘宝商品，不只是打开搜索页，还涉及：

- 开放平台权限
- 淘宝客转链
- 商品搜索 API
- 价格库存同步
- 优惠券和佣金追踪
- 合规和风控

解决办法：

PoC 阶段先用淘宝移动端搜索链接，保证完整跳转体验。后续商业化再接淘宝开放平台或阿里妈妈淘宝客能力。

### 4.9 分享图跨浏览器差异

问题表现：

不同手机浏览器对 Web Share、文件分享、下载能力支持不一致。

解决办法：

- 优先尝试 `navigator.share`
- 支持文件分享时调用系统分享
- 不支持时降级为 PNG 下载
- Canvas 生成逻辑放在本地，不依赖后端

### 4.10 PWA 在局域网 HTTP 下受限

问题表现：

本地开发时，电脑 `localhost` 可以注册 service worker，但手机通过局域网 IP 访问 HTTP 地址时，部分浏览器不会注册。

原因：

service worker 需要安全上下文，通常要求 HTTPS；`localhost` 是特例，但局域网 IP 的 HTTP 不是安全上下文。

解决办法：

- 注册逻辑中判断 `window.isSecureContext`
- 非安全上下文下跳过注册，避免控制台报错
- 部署到 HTTPS 后即可正常使用 PWA 能力

## 5. 当前架构总结

当前项目可以理解为四个核心模块：

```txt
扫描入口层
  UploadCard / ScanAnimation

AI 编排层
  /api/scan-object / dashscope / prompt

结果展示层
  /result / ScanResultPanel / ShareResultButton

本地业务层
  storage / serviceMap / history / service pages
```

这种拆分让 PoC 足够轻，但主要职责已经分开：

- 首页负责扫描
- API 负责识别
- 结果页负责展示
- 服务映射负责推荐
- 本地存储负责历史和当前结果恢复
